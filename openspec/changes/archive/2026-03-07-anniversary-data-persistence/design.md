## Context

기념일 기능은 현재 계산 유틸과 화면 렌더링 중심으로 구현되어 있으며, 사용자별 원본 데이터의 서버 저장 계층이 없다. 이 상태에서는 멀티 디바이스 동기화, 기념일 변경 이력, 권한 검증, 알림 확장 시 매번 로컬 상태에 의존해야 하므로 확장성이 떨어진다.

이번 변경은 서버 우선 구조로 기념일 데이터를 정규화해 저장하고, 달력 월 뷰에서 시험 일정과 기념일을 하나의 API 응답으로 결합하는 것을 목표로 한다.

## Goals / Non-Goals

**Goals:**
- 사용자별 기념일 데이터 영구 저장 모델을 도입한다.
- 기념일 CRUD API를 표준 응답 포맷으로 제공한다.
- 월 단위 달력 조회에서 일정 + 기념일을 통합 반환한다.
- 사용자 소유권 기반 접근 제어와 감사 로그를 보장한다.

**Non-Goals:**
- 커플 간 동시 편집 충돌 자동 병합
- 고급 반복 규칙 엔진(RRULE 완전 지원)
- 푸시 알림 발송 채널 구현

## Decisions

### 1) 기념일 저장소를 별도 테이블로 분리한다
- 결정: `anniversaries` 테이블에 `id/user_id/name/base_date/rule_type/rule_value/is_active`를 저장한다.
- 이유: 일정 이벤트와 분리해야 계산 규칙 확장과 권한 정책 적용이 단순해진다.
- 대안:
  - `exam_events`에 혼합 저장: 도메인 의미가 섞여 유지보수성이 낮아진다.

### 2) CRUD API는 서버 표준 응답 규약을 그대로 따른다
- 결정: `success/data/errorCode/errorMessage/requestId` 포맷을 유지한다.
- 이유: 기존 서버 공통 계층과 클라이언트 에러 처리 일관성을 유지한다.
- 대안:
  - 기념일 전용 응답 포맷: 클라이언트 분기 증가 및 관측 복잡도 상승.

### 3) 월 뷰 조회는 서버에서 결합한다
- 결정: 월 범위를 입력받아 `exam_events + anniversaries`를 서버에서 병합 후 정렬해 반환한다.
- 이유: 클라이언트별 병합 구현 중복을 제거하고 데이터 정합성을 높일 수 있다.
- 대안:
  - 클라이언트 병합: 채널별 로직 중복과 누락 가능성 증가.

### 4) 접근 제어는 user_id 소유권 기준으로 강제한다
- 결정: RLS + 핸들러 검증으로 본인 데이터만 CRUD 가능하도록 제한한다.
- 이유: 역할(A/B)과 별개로 개인 데이터 경계를 명확히 해야 한다.
- 대안:
  - 애플리케이션 레벨 체크만 사용: 우회 요청에 취약해진다.

## Risks / Trade-offs

- [월 뷰 결합 쿼리 성능 저하] -> Mitigation: `user_id + date` 인덱스와 월 범위 쿼리 제한 적용
- [규칙 파싱 오류로 잘못된 기념일 생성] -> Mitigation: 서버 입력 검증(Zod)과 경계값 테스트 추가
- [RLS 오설정으로 데이터 누출 위험] -> Mitigation: 소유권 기반 정책 테스트 케이스 필수화
- [클라이언트-서버 시간대 불일치] -> Mitigation: 저장은 UTC 기준, 표시만 로컬 시간대로 변환

## Migration Plan

1. `infra/migrations`에 `anniversaries` 테이블 및 인덱스/RLS 정책을 추가한다.
2. `services/functions`에 기념일 CRUD 및 월 뷰 조회 핸들러를 추가한다.
3. 기존 달력 조회 경로를 신규 API로 전환하고 로컬 fallback을 유지한다.
4. 감사 로그에 기념일 생성/수정/삭제 이벤트를 기록한다.
5. 테스트(권한/검증/월 뷰 결합)를 추가한 뒤 배포한다.

Rollback 전략:
- 신규 핸들러를 feature flag로 제어해 즉시 비활성화 가능하게 한다.
- 테이블은 read-only 모드로 전환한 뒤 후속 마이그레이션으로 정리한다.
