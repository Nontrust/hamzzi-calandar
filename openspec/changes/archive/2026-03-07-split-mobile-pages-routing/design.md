## Context

현재 모바일 웹 앱은 `app/index.tsx`에서 로그인 화면과 메인 화면을 조건부 렌더링으로 함께 처리한다. 이 구조는 다음 문제를 만든다.
- 파일 단위 책임이 과도하게 커져 기능 추가/수정 시 영향 범위 파악이 어렵다.
- 인증 상태 기반 접근 제어가 컴포넌트 내부 조건문에 묶여 라우팅 정책이 불명확하다.
- 페이지 분리(로그인 전용 개선, 홈 탭 확장, 설정 페이지 추가) 시 중복 코드가 늘어날 가능성이 높다.

이번 변경은 Expo Router를 기준으로 인증 전/후 화면을 분리해, 페이지 경계와 접근 규칙을 명확히 한다. 기존 도메인 로직(auth client, anniversary/calendar/permissions)은 유지하고 화면 책임만 재배치한다.

## Goals / Non-Goals

**Goals:**
- 로그인 페이지와 홈 페이지를 파일/라우트 단위로 분리한다.
- 인증 상태에 따른 라우팅 규칙(미인증: 로그인, 인증: 홈)을 일관되게 적용한다.
- 기존 기능(로그인, 역할 전환, 기념일 CRUD, 권한 요청, 오류 안내)을 회귀 없이 유지한다.
- 기존 UI 토큰/브랜딩 자산 적용을 페이지 분리 후에도 동일하게 유지한다.

**Non-Goals:**
- 서버 API/데이터 모델 변경
- 인증 정책(계정/권한 체계) 변경
- 화면 디자인 전면 재구성
- 탭 네비게이션/딥링크 설계 확장

## Decisions

1. **라우트 그룹 분리 구조 채택**
- 결정: `app/(auth)/login.tsx`, `app/(app)/index.tsx` 구조와 그룹별 `_layout.tsx`를 사용한다.
- 이유: 인증 전/후 페이지를 디렉터리 레벨에서 분리하면 책임 경계가 명확해지고 확장 시 충돌이 줄어든다.
- 대안: 기존 `app/index.tsx` 단일 파일 유지. 기각 이유: 유지보수성 저하와 회귀 위험 지속.

2. **인증 가드 위치를 레이아웃 계층으로 이동**
- 결정: 루트 또는 그룹 레이아웃에서 세션 확인 후 `router.replace`로 안전 경로를 강제한다.
- 이유: 각 페이지에서 중복 가드를 구현하지 않아도 되고 접근 정책을 중앙화할 수 있다.
- 상태 전이 규칙:
  - 앱 시작: `loading -> authenticated | unauthenticated`
  - 미인증 접근: `unauthenticated -> /login`
  - 로그인 성공: `authenticated -> /`
  - 로그아웃: `unauthenticated -> /login`
- 대안: 페이지별 useEffect 가드. 기각 이유: 중복과 레이스 컨디션 증가.

3. **데이터 모델 무변경 + 화면 전용 뷰모델 유지**
- 결정: auth session 및 calendar/anniversary 데이터 구조는 유지하고, 페이지별로 필요한 상태만 분리한다.
- 이유: 기능 회귀를 최소화하고 변경 범위를 라우팅/구조로 제한한다.
- 데이터 변경점: 없음 (클라이언트 상태 보관 위치만 재배치).

4. **실패 처리 정책 유지 및 페이지 분리 대응**
- 결정: API/권한 실패 시 현재 페이지를 유지하고 사용자 안내 문구를 갱신한다. 치명적 예외 시 로그인 폴백 경로를 제공한다.
- 이유: 사용자의 작업 맥락 유지가 우선이며, 페이지 이동으로 인한 상태 손실을 최소화해야 한다.
- 예외 처리:
  - 세션 조회 실패: 로그인 페이지로 폴백
  - 권한 요청 실패: 상태값 `error` 및 안내 문구 유지
  - API 실패: 마지막 성공 데이터 유지 + 재시도 가능 상태 보장

## Risks / Trade-offs

- [Risk] 초기 렌더에서 라우트 리다이렉트 타이밍이 겹쳐 깜빡임 발생 가능 → [Mitigation] 인증 로딩 상태(`isBootstrapping`) 동안 스플래시/빈 컨테이너 렌더 후 단일 리다이렉트 수행
- [Risk] 기존 단일 파일에서 함수 분리 시 참조 누락 가능 → [Mitigation] 공통 훅/핸들러를 페이지별로 명시 이동하고 타입체크/수동 시나리오 검증 수행
- [Trade-off] 라우트 파일 수 증가로 초기 구조가 복잡해짐 → [Mitigation] 그룹 레이아웃 규칙을 문서화하고 네이밍 규칙을 고정

## Migration Plan

1. `app` 디렉터리에 auth/app 그룹 라우트 파일 생성
2. 기존 `index.tsx`에서 로그인 UI와 홈 UI를 분리 이관
3. 루트/그룹 레이아웃에 인증 가드 로직 적용
4. 로그인/로그아웃 리다이렉트 흐름 검증
5. 타입체크 및 핵심 수동 시나리오(로그인, 역할 전환, 기념일 CRUD, 권한 요청) 점검

Rollback 전략:
- 라우트 분리 커밋 단위로 되돌리면 기존 단일 파일 구조로 복구 가능
- 문제 발생 시 `app/index.tsx` 단일 렌더링 구조로 즉시 롤백

- 로그인 페이지 경로를 `/login` 단일로 유지할지, 추후 `/auth/login` 네임스페이스로 확장할지
