## Context

현재 코드베이스는 `services/functions`를 이미 사용하고 있지만, 기능별로 서버 경계 규칙이 일관되게 명세되어 있지 않다. 특히 인증/권한 검증, 외부 토큰 수명주기, 재시도 정책, 공통 에러 포맷이 기능별로 흩어지면 차후 캘린더/AI/기념일 확장에서 중복 구현과 보안 누락이 발생할 수 있다.

핵심 이해관계자는 다음과 같다.
- 사용자 A/B: 기능 실패 시 복구 가능하고 의미가 명확한 응답을 기대한다.
- 운영자: 실패 원인 추적, 재시도 제어, 감사 로그가 필요하다.
- 개발자: 클라이언트/서버 책임이 분리된 구조에서 기능을 빠르게 확장해야 한다.

## Goals / Non-Goals

**Goals:**
- Edge Functions 공통 API 계층(BFF) 규약을 정의한다.
- 인증/권한/감사 로그를 서버단 정책으로 통합한다.
- 외부 연동 토큰의 저장/갱신/폐기 규칙을 고정한다.
- 수집/동기화 작업의 재시도/백오프/실패 복구 상태 전이를 표준화한다.
- 클라이언트는 표시/입력에 집중하고, 서버는 보안/연동/일관성 책임을 갖도록 경계를 명확히 한다.

**Non-Goals:**
- 멀티 리전/멀티 클라우드 분산 아키텍처 도입
- Kafka 등 별도 메시지 브로커 기반 대규모 큐 시스템 도입
- 기능 전체를 마이크로서비스로 분리

## Decisions

### 1) 서버 API 표준 응답 포맷을 단일화한다
- 결정: 모든 서버 응답은 `success/data/errorCode/errorMessage/requestId` 구조를 사용한다.
- 이유: 클라이언트 복구 흐름과 관측(로그 상관관계)을 단순화할 수 있다.
- 대안:
  - 핸들러별 임의 포맷 유지: 확장 시 에러 처리 중복이 커진다.

### 2) 인증/권한 검증은 서버 진입점에서 선행한다
- 결정: Edge Function 진입 시점에 사용자 신원 확인과 역할 권한 확인을 수행하고, 실패 시 표준 에러 코드로 즉시 반환한다.
- 이유: 기능별 비즈니스 코드에서 권한 체크 누락 위험을 줄일 수 있다.
- 대안:
  - 각 기능 내부에서 분산 체크: 누락/불일치 가능성이 높다.

### 3) 외부 토큰은 암호화 저장과 만료 전 갱신을 기본 정책으로 한다
- 결정: OAuth 토큰은 암호화된 저장소에 보관하고, 만료 임계값 이전 선갱신을 시도한다.
- 이유: 사용자 체감 실패를 줄이고 보안 사고면을 축소할 수 있다.
- 대안:
  - 요청 시점 갱신만 수행: 트래픽 집중 시 실패율이 상승한다.

### 4) 재시도 정책은 상태 전이 + 백오프로 표준화한다
- 결정: `pending -> failed -> pending(retry)` 전이를 유지하면서 지수 백오프와 최대 재시도 횟수를 적용한다.
- 이유: 외부 API 일시 장애에 탄력적으로 대응하고 무한 재시도를 방지할 수 있다.
- 대안:
  - 고정 간격 무제한 재시도: 비용/혼잡/장애 전파 위험이 크다.

### 5) 클라이언트/서버 책임 경계를 명시한다
- 결정: 클라이언트는 입력/렌더링/로컬 계산, 서버는 외부 연동/보안/상태 일관성 담당으로 분리한다.
- 이유: 모바일/웹 채널 확장 시 중복 구현 없이 일관성을 유지할 수 있다.
- 대안:
  - 클라이언트 중심 연동: 토큰 노출과 권한 우회 위험이 증가한다.

## Risks / Trade-offs

- [서버 공통 계층 도입으로 초기 개발 속도 저하] -> Mitigation: 최소 골격(인증/에러/추적)부터 적용하고 점진 확장
- [토큰 암호화/갱신 로직 복잡도 증가] -> Mitigation: 토큰 모듈을 단일 책임으로 분리하고 테스트 케이스 선행 작성
- [재시도 정책 오설정으로 지연 증가] -> Mitigation: 최대 재시도 횟수와 상한 백오프를 운영 설정으로 분리
- [권한 정책 강화로 기존 클라이언트 호출 실패 가능] -> Mitigation: 에러 코드 매핑 문서 제공 및 단계적 강제 적용

## Migration Plan

1. 서버 공통 응답/에러/요청 추적 유틸을 `services/functions`에 도입한다.
2. 인증/권한 미들웨어를 추가하고 기존 핸들러에 단계적으로 적용한다.
3. 외부 토큰 저장 스키마(암호화 필드/만료시각/갱신시각)를 마이그레이션에 반영한다.
4. 재시도/백오프 정책을 작업 실행 계층에 적용하고 상태 전이 로깅을 추가한다.
5. 운영 문서에 에러 코드와 복구 절차를 반영한다.

Rollback 전략:
- 공통 미들웨어 적용 전 핸들러를 feature flag로 전환 가능하게 유지한다.
- 토큰 스키마는 이전 필드를 일정 기간 병행 유지해 되돌림을 허용한다.
- 재시도 정책은 설정값 롤백으로 즉시 완화할 수 있게 한다.
