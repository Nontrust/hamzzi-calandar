## Context

현재 앱은 공통 셸(사이드바/헤더/모바일 드로어)과 메뉴 라우트 골격을 갖췄지만, 기능이 한 페이지에 집중되어 메뉴 의미와 실제 동작이 일치하지 않는 상태다. 이번 변경은 기능을 메뉴 기준 페이지로 분리해 유지보수성과 확장성을 높이는 데 목적이 있다.

제약사항:
- 기존 인증 가드와 세션 흐름은 유지해야 한다.
- 기존 핵심 기능(기념일 CRUD, 월간 일정 조회, 권한 요청, 로그아웃)은 회귀 없이 동작해야 한다.
- 모바일과 웹에서 동일한 메뉴 구조를 제공해야 한다.

## Goals / Non-Goals

**Goals:**
- 메뉴 기준으로 홈/기념일/일정/설정 페이지를 분리한다.
- 공통 셸은 페이지 문맥(헤더 타이틀/보조 문구)을 반영한다.
- 모바일 드로어 메뉴와 데스크톱 사이드바 메뉴의 동작을 일치시킨다.

**Non-Goals:**
- 신규 API 추가
- 디자인 시스템 전면 개편
- 정보구조 전면 재설계

## Decisions

1. **레이아웃은 유지, 콘텐츠를 페이지로 분리**
- 결정: `app/(app)/_layout.tsx` 공통 셸은 유지하고, 기능 콘텐츠만 각 라우트 파일로 분리한다.
- 이유: 인증 가드/공통 UI 중복을 줄이고 기능 분리를 빠르게 달성한다.

2. **메뉴 메타데이터 중심 네비게이션**
- 결정: 메뉴 라벨/경로/헤더 보조문구를 단일 배열로 관리한다.
- 이유: 사이드바/드로어/헤더 문구를 하나의 소스로 동기화할 수 있다.

3. **기능 이관은 단계적**
- 결정: 1차로 각 페이지에 핵심 기능 흐름을 배치하고, 세부 UI 고도화는 후속 변경으로 분리한다.
- 이유: 회귀 리스크를 낮추면서 빠르게 구조를 정리한다.

## Risks / Trade-offs

- [리스크] 기능 분리 중 상태 관리 중복 증가
  - 완화: 역할/세션 관련 로직을 공통 훅으로 단계적 통합
- [리스크] 모바일 드로어 오버레이 상호작용 문제
  - 완화: 오버레이 탭 닫기, 메뉴 선택 시 닫기 동작 강제
- [트레이드오프] 단기적으로 페이지 UI 완성도가 균일하지 않을 수 있음
  - 완화: 기능 우선 분리 후 시각 고도화 작업 분리

## Migration Plan

1. `_layout` 메뉴 메타데이터 및 라우팅 연결
2. `anniversaries/schedule/settings` 페이지 파일 생성
3. 기존 `index` 기능 중 일부를 각 페이지 책임에 맞춰 이동
4. 타입체크/라우팅 테스트/수동 QA 체크리스트 갱신

Rollback 전략:
- 레이아웃 및 신규 페이지 파일 단위로 되돌리면 기존 단일 페이지 구조로 복귀 가능
