## Context

현재 서비스는 역할 선택 기반으로만 사용자 컨텍스트를 다루고 있어, 계정 인증 없이 핵심 기능 접근이 가능한 상태다. 이번 변경은 `user-auth`를 신규 도입하고, 기존 `privacy-and-permissions`, `server-api-foundation`, `user-role-selection` 요구사항을 인증 기반으로 재정렬하는 작업이다.

기술 제약은 다음과 같다.
- 기존 functions 응답 규약(`success/data/errorCode/errorMessage/requestId`)과 호환되어야 한다.
- 캘린더 외부 연동 실패와 인증 실패를 분리해야 한다.
- 데모 계정(나햄찌, 햄찌 메이트)은 개발/테스트 환경에서 초기 시드로 제공하되, 비밀번호 평문 하드코딩은 금지한다.

## Goals / Non-Goals

**Goals:**
- 아이디/비밀번호 로그인, 로그아웃, 세션 검증의 최소 인증 흐름을 구축한다.
- 인증 필요 API 진입 전에 공통 인증 게이트를 강제한다.
- AUTH 계열 오류 코드를 공통 응답 규약에 맞춰 표준화한다.
- 역할 선택은 로그인 사용자 컨텍스트를 기반으로 동작하도록 상태 전이를 명확히 한다.

**Non-Goals:**
- 소셜 로그인, 비밀번호 재설정 메일, 2FA, 회원가입 UX
- 다중 기기 세션 동기화/강제 로그아웃
- 운영 환경 비밀관리 시스템 전체 교체

## Decisions

### 1) 인증 저장소를 `users` + `sessions` 구조로 분리한다
- 결정: 사용자 기본 정보(`users`)와 세션(`sessions`)을 분리하고, 비밀번호는 해시(`password_hash`)만 저장한다.
- 이유: 세션 만료/폐기 관리, 감사 로깅, 추후 토큰 전략 확장에 유리하다.
- 대안:
  - JWT 무상태 단독: 서버 폐기 제어/강제 로그아웃 대응이 어려워 MVP에서 제외.
  - 단일 테이블 혼합: 상태 전이가 불명확해 유지보수 리스크 증가.

### 2) 상태 전이를 명시적으로 정의한다
- 결정: `anonymous -> authenticated -> role_selected` 상태 전이를 사용한다.
- 규칙:
  - 로그인 성공 시 `authenticated`
  - 역할 선택 완료 시 `role_selected`
  - 로그아웃/세션 만료 시 `anonymous`로 복귀
- 이유: 기존 역할 선택 기능과 인증 기능을 느슨하게 결합하면서 권한 경계를 명확히 하기 위함.
- 대안:
  - 역할을 세션 생성 시 강제: 유연성이 떨어지고 기존 UX와 충돌.

### 3) 인증 실패와 외부 연동 실패를 분리 처리한다
- 결정: AUTH 실패는 `AUTH_REQUIRED`, `AUTH_INVALID_CREDENTIALS`, `AUTH_SESSION_EXPIRED` 등으로 응답하고, 캘린더 연동 실패는 기존 연동 오류 코드로 별도 처리한다.
- 이유: 사용자 안내 문구 및 재시도 전략이 다르기 때문이다.
- 대안:
  - 단일 `INTERNAL_ERROR`: 원인 식별 불가, UX 저하.

### 4) 데모 계정은 시드 절차로 주입한다
- 결정: 데모 계정(나햄찌/햄찌 메이트)은 마이그레이션 또는 시드 스크립트에서 생성하며, 비밀번호 원문은 환경변수 입력값으로 해시 후 저장한다.
- 이유: 저장소/문서 평문 노출 방지.
- 대안:
  - 코드 상수 하드코딩: 보안 리스크가 커서 제외.

## Risks / Trade-offs

- [세션 저장소 장애 시 로그인 전체 영향] -> Mitigation: 세션 조회 실패 시 명시적 장애 코드 + 헬스체크/재시도 정책 적용
- [브루트포스 로그인 시도 증가] -> Mitigation: 계정/아이피 기준 rate limit, 실패 횟수 임계치 잠금
- [인증 도입으로 기존 화면 접근 회귀] -> Mitigation: 보호 라우트 회귀 테스트 + 비로그인 시 리다이렉트 테스트 추가
- [시드 계정 운영 혼입] -> Mitigation: 환경 분리(dev/test 전용), 운영 배포 파이프라인에서 시드 비활성화

## Migration Plan

1. DB 스키마 추가: `users`, `sessions`(및 인덱스/만료 컬럼) 생성
2. 시드 절차 추가: 데모 계정 2종 생성(비밀번호 해시 저장)
3. 서버 핸들러 추가: login/logout/session-validate
4. 보호 API 공통 인증 게이트 적용
5. 클라이언트 로그인 화면/세션 스토어/보호 라우트 적용
6. 롤백 전략:
   - 기능 플래그로 인증 게이트 비활성화
   - 신규 인증 엔드포인트 트래픽 차단
   - 세션 테이블은 read-only 유지 후 후속 정리
