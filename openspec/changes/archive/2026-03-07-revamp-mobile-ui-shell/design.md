## Context

현재 `apps/mobile-web/app/index.tsx`는 기능 검증과 UI 렌더링 로직이 한 파일에 혼재되어 있어 스타일 재사용과 화면 일관성 유지가 어렵다. 또한 `openspec/statics`의 브랜드 자산이 일부 화면에만 반영되어 앱 아이덴티티가 약하다. 이번 변경은 기능 로직을 유지하면서 UI 계층을 정리해, 로그인 화면과 메인 화면을 동일한 디자인 시스템으로 묶는 것을 목표로 한다.

제약사항은 다음과 같다.
- 기존 인증/역할/기념일/권한 요청 동작은 회귀 없이 유지해야 한다.
- 서버 API 스키마 변경 없이 클라이언트 렌더링 계층만 개선한다.
- 실패 시 크래시 대신 안내 메시지와 마지막 정상 상태를 유지해야 한다.

## Goals / Non-Goals

**Goals:**
- 공통 스타일 토큰(`theme`) 도입으로 색상/간격/타이포/반경 정의를 중앙화한다.
- 로그인/메인 화면 UI를 카드 기반 앱 쉘로 통일한다.
- 브랜드 자산(로고/BI)을 일관 배치한다.
- 외부 연동 실패 시 상태 전이 규칙을 명확히 적용한다.

**Non-Goals:**
- 인증 정책, API 응답 구조, DB 모델 변경
- 신규 화면 라우팅 추가
- 네이티브 플랫폼 전용 UI 분기

## Decisions

1. **공통 스타일 파일 분리**
- 결정: `apps/mobile-web/app/src/ui/theme.ts`(토큰)와 `appStyles.ts`(StyleSheet)를 생성해 화면 스타일을 분리한다.
- 이유: 한 파일 내 대규모 스타일 정의로 인한 유지보수 비용을 줄이고 재사용성을 확보한다.
- 대안: `index.tsx` 내 인라인 스타일 유지. 기각 이유는 중복과 회귀 위험이 큼.

2. **화면 구조는 기존 단일 스크린 유지 + UI 컴포넌트 경량 분리**
- 결정: 라우팅 구조는 변경하지 않고, 버튼/카드/상태배지 수준만 경량 컴포넌트로 분리한다.
- 이유: 기능 회귀 위험을 낮추면서 구조 개선 효과를 빠르게 얻을 수 있다.
- 대안: 화면 전면 분할(로그인/메인 별도 파일). 기각 이유는 이번 MVP 범위를 초과.

3. **실패 처리 상태 전이 규칙 고정**
- 결정: 네트워크/권한 실패 시 `apiNotice` 또는 권한 상태 텍스트를 갱신하고, 리스트 데이터는 마지막 성공 상태를 유지한다.
- 이유: 사용자 작업 연속성을 보장하고 빈 화면/크래시를 방지한다.
- 상태 전이:
  - 성공: `idle|error -> synced`
  - 실패: `synced -> error(notice만 갱신, 데이터 유지)`
  - 재시도 성공: `error -> synced`
- 대안: 실패 시 데이터 초기화. 기각 이유는 사용성 저하.

4. **데이터 모델 변경 없음, 뷰모델만 정리**
- 결정: 서버/스토리지 스키마는 유지하고, 화면 계산값(`monthSummary`, notice)을 뷰모델 성격으로 분리 유지한다.
- 이유: 범위를 UI 개편에 집중하고 테스트 영향을 최소화한다.

## Risks / Trade-offs

- [리스크] 스타일 분리 과정에서 버튼/카드 상호작용 스타일 누락 가능 → [완화] 주요 액션(로그인/역할전환/CRUD/권한요청) 수동 시나리오 테스트 수행
- [리스크] 브랜드 이미지 경로 오타로 번들 실패 가능 → [완화] 정적 자산 `require` 경로 고정 및 웹 실행 확인
- [트레이드오프] 단일 스크린 구조 유지로 완전한 컴포넌트 아키텍처는 미도입 → [완화] 다음 변경에서 페이지 단위 분리 여지를 남김

## Migration Plan

1. 공통 토큰/스타일 파일 추가
2. `index.tsx`에서 스타일 참조 전환 및 구조 정리
3. 실패 처리 문구/상태 유지 동작 회귀 점검
4. 웹 실행(`npm --workspace apps/mobile-web run web -- --offline`)으로 UI 및 기능 검증

Rollback 전략:
- UI 파일 변경만 되돌리면 기능 레이어에는 영향이 없도록 분리한다.
- 문제 발생 시 `index.tsx`와 신규 UI 파일만 이전 커밋으로 복원한다.
