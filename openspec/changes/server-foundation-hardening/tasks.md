## 1. 서버 공통 API 계층 정비

- [x] 1.1 Edge Function 공통 응답 포맷(`success/data/errorCode/errorMessage/requestId`) 유틸을 구현한다 (검증: 정상/실패 응답이 동일 스키마로 반환된다).
- [x] 1.2 인증/권한 검증 미들웨어를 추가하고 기존 핸들러 진입점에 적용한다 (검증: 권한 없는 호출이 비즈니스 로직 실행 전에 차단된다).
- [x] 1.3 요청 추적 ID 생성/전달 로직을 공통화한다 (검증: 에러 응답과 로그에 동일 requestId가 남는다).

## 2. 외부 토큰 수명주기 구현

- [x] 2.1 외부 연동 토큰 저장 스키마(암호화 값, 만료시각, 갱신시각)를 마이그레이션에 반영한다 (검증: 토큰 레코드 생성 시 필수 필드가 저장된다).
- [x] 2.2 토큰 선갱신 임계값 정책과 갱신 함수(성공/실패 분기)를 구현한다 (검증: 만료 임계 구간 토큰이 자동 갱신된다).
- [x] 2.3 갱신 실패 시 재시도 가능 상태 및 재연결 필요 상태 전이를 구현한다 (검증: 연속 실패 시 사용자 재연결 안내 코드가 반환된다).

## 3. 작업 재시도 오케스트레이션

- [x] 3.1 수집/동기화 작업에 지수 백오프 재시도 정책을 적용한다 (검증: 일시 실패 시 재시도 간격이 증가한다).
- [x] 3.2 작업 상태 전이(`pending/failed/retrying/synced`)를 표준화하고 기록한다 (검증: 성공/실패/재시도 흐름에서 상태 기록이 일관된다).
- [x] 3.3 최대 재시도 횟수 초과 보호를 적용한다 (검증: 한도 초과 시 자동 재시도가 중단되고 운영 확인 상태가 기록된다).

## 4. 관측/문서/회귀 검증

- [x] 4.1 권한 거부/민감 API 접근 실패 감사 로그를 추가한다 (검증: 거부 이벤트에 requestId와 사유가 기록된다).
- [x] 4.2 서버 실패 코드와 복구 안내 매핑 문서를 작성한다 (검증: 운영 문서만으로 실패 코드 대응 절차를 확인할 수 있다).
- [x] 4.3 회귀 테스트를 추가해 기존 캘린더/면접 흐름과 신규 서버 규약이 함께 유지되는지 확인한다 (검증: 테스트 스위트가 모두 통과한다).
