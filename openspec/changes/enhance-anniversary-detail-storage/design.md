## Context

기념일 기능이 단순 이름/날짜 중심으로 동작해 실제 사용 시 필요한 상세 정보(카테고리, 메모, 반복 규칙, 알림 여부)를 담기 어렵다.  
이번 변경은 모바일 앱, API 핸들러, 저장소 계층, 마이그레이션까지 영향을 주는 cross-cutting 변경이다.

## Goals / Non-Goals

**Goals:**
- 기념일 상세 정보 저장 구조를 정의하고 CRUD 흐름에 반영한다.
- 기존 데이터와 호환 가능한 기본값/마이그레이션 전략을 제공한다.
- 월간/일간 조회 응답에서 상세 필드를 안정적으로 제공한다.
- 외부 공휴일/캘린더 연동 실패와 무관하게 기념일 조회를 유지한다.

**Non-Goals:**
- 인증/권한 체계 변경
- 신규 외부 서비스 연동 추가
- 캘린더 UI 대규모 리디자인

## Decisions

1. 상세 필드 확장 범위 고정
- `name`, `baseDate` 외에 `category`, `note`, `reminderEnabled`, `reminderOffsetDays`, `repeatRule`(ruleType/ruleValue)를 저장 필드로 정의한다.
- 대안: 자유 JSON 필드 1개에 모두 저장.
- 채택 이유: 타입 안정성과 서버 검증 일관성이 높고 조회/필터 확장이 쉽다.

2. 서버 우선 저장 + 클라이언트 폴백 유지
- 온라인 시 서버 API를 기준으로 저장/조회하고, 실패 시 기존 로컬 폴백 전략을 유지한다.
- 대안: 서버 저장 강제(실패 시 에러만 노출).
- 채택 이유: 현재 오프라인/불안정 네트워크 상황에서도 UX를 유지할 수 있다.

3. 하위 호환 마이그레이션
- 기존 레코드에는 기본값(`category=general`, `reminderEnabled=false`, `note=''`)을 적용한다.
- 서버 마이그레이션은 nullable → backfill → 필요 시 not null 순서로 진행한다.
- 대안: 스키마 강제 변경 후 클라이언트 일괄 업데이트.
- 채택 이유: 배포 중간 구간의 호환성 리스크를 줄인다.

4. 입력 검증을 서버 단일 소스로 정리
- 날짜 형식, 메모 길이, 오프셋 범위, rule 조합 검증을 서버에서 최종 검증한다.
- 클라이언트는 UX 수준 사전 검증만 수행한다.
- 대안: 클라이언트/서버 동일 검증 이중 구현.
- 채택 이유: 규칙 변경 시 서버 중심 관리가 안전하다.

## Risks / Trade-offs

- [Risk] 상세 필드 증가로 API 페이로드가 커짐 → Mitigation: 목록/상세 응답 필드를 분리하거나 필요한 최소 필드만 전송
- [Risk] 기존 데이터의 기본값 적용 오류 → Mitigation: 마이그레이션 후 샘플 데이터 검증 쿼리와 회귀 테스트 수행
- [Risk] 폴백 저장과 서버 저장 간 불일치 → Mitigation: 동기화 시 서버 우선 merge 규칙과 충돌 로그 추가
- [Risk] 반복 규칙 검증 누락 → Mitigation: 서버 단위 테스트로 ruleType/ruleValue 조합 케이스 고정

## Migration Plan

1. DB 스키마 확장 마이그레이션 추가
2. 서버 `anniversaryStore`/`handlers`에서 입력/출력 타입 확장
3. 모바일 `anniversaryClient` 요청/응답 타입 및 폴백 저장 포맷 확장
4. 기념일 생성/수정 UI에서 상세 입력 필드 반영
5. 월간/일간 조회 화면에서 상세 정보 노출 규칙 적용
6. 타입체크/테스트/수동 QA 후 배포
