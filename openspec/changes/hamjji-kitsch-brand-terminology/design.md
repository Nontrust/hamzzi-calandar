## Context

이번 변경은 기능 로직 자체를 바꾸는 것이 아니라, 사용자에게 노출되는 용어와 카피 체계를 "햄찌 세계관" 기반으로 재설계하는 작업이다. 핵심은 기존의 내부 기능 식별자(예: `user-role-selection`, `study-execution`)는 유지하되, UI/리포트/캘린더 제목 등 사용자 접점 문구를 키치 톤으로 일관되게 매핑하는 것이다.

현재 제약은 다음과 같다.
- 기존 스펙/코드의 기능 의미를 훼손하면 안 된다.
- 외부 연동(캘린더, API, AI 리포트) 문자열은 길이/가독성 제한을 고려해야 한다.
- 용어 톤은 귀엽고 캐릭터성 있어야 하지만, 저속/비하 표현은 금지해야 한다.
- 향후 용어 수정이 쉬워야 하므로 하드코딩이 아닌 중앙 관리가 필요하다.

## Goals / Non-Goals

**Goals:**
- 홈 화면 제품명을 `햄찌의 뽀짝 하루 캘린더`로 변경해 하루를 함께하는 캘린더 맥락을 즉시 인지하게 한다.
- 화면 노출용 용어를 중앙 용어집(`hamjji-brand-lexicon`)으로 정의한다.
- 내부 식별자와 외부 노출명을 분리하는 매핑 규칙을 도입한다.
- 사용자 A/B, 세션, 보상, 일정, 면접 관련 핵심 문구를 햄찌 톤으로 일괄 치환한다.
- 금지어/주의어/허용 톤 가이드를 문서화해 카피 품질을 유지한다.
- 연동 실패/오류 메시지에서도 의미 명확성을 유지하는 문구 정책을 정의한다.
- 기념일 계산기 기준일/반복주기 규칙을 정의하고 달력 내 기념일 표기 방식을 통일한다.

**Non-Goals:**
- 다국어(i18n) 전체 프레임워크 도입
- 개인별 톤 실험(A/B 테스트) 자동화
- 자동 카피 생성 AI 파이프라인 구축
- 기능 플로우/권한 로직/데이터 모델 변경

## Decisions

### 1) 내부 식별자와 노출명을 분리한다
- 결정: 코드/DB/이벤트 키는 기존 식별자를 유지하고, UI 노출은 별도 레이블 맵에서 가져온다.
- 이유: 기능 안정성과 브랜드 실험을 분리하면 변경 리스크가 낮다.
- 대안:
  - 식별자 자체를 햄찌 용어로 변경: 코드 가독성/호환성 저하 가능성이 크다.

### 2) 용어집 단일 소스(`hamjji-brand-lexicon`)를 신설한다
- 결정: 카테고리별(역할/세션/보상/일정/면접/오류) 용어를 한 파일 또는 한 모듈에서 관리한다.
- 이유: 화면별 하드코딩을 막고 QA 검수 포인트를 명확히 한다.
- 대안:
  - 화면마다 개별 문자열 관리: 누락/불일치 가능성이 높다.

### 3) 노출명 데이터 모델 규칙을 정의한다
- 결정: 각 용어 항목은 `id`, `defaultLabel`, `kitschLabel`, `description`, `usageScope`, `safetyLevel` 필드를 갖는다.
- 이유: 변경 이력 관리와 톤 안전성 점검이 가능하다.
- 대안:
  - 단순 key-value 구조: 용어 의도/범위를 추적하기 어렵다.

### 4) 톤 안전 가이드를 명시한다
- 결정: "귀엽고 장난스러운 톤"은 허용하되, 저속어/비하어/대상 비하형 별칭은 금지한다.
- 이유: 개인화 브랜딩을 유지하면서 서비스 품질과 신뢰를 보장하기 위함.
- 대안:
  - 완전 자유 카피: 일관성과 안정성이 무너질 수 있다.

### 5) 연동 메시지 이중 원칙을 적용한다
- 결정: 외부 연동(캘린더/API/AI) 메시지는 "브랜드 톤 + 기능 의미"를 동시 표기한다.
- 예: "일정 동기화 잠깐 삐끗 (일정 동기화 실패: 다시 시도해줘)"
- 이유: 귀여운 톤만 강조하면 오류 원인 파악이 어려워질 수 있다.
- 대안:
  - 순수 키치 문구만 사용: 운영/지원 상황에서 해석성이 떨어진다.

### 6) 적용 우선순위를 정의한다
- 1순위: 역할 선택, 홈/세션 핵심 카피, 보상 카드 명칭
- 2순위: 일정 이벤트 제목 템플릿, 면접 리포트 헤더
- 3순위: 오류/예외 메시지 세부 문구 정교화, 기념일 계산/표기

### 7) 기념일 계산 규칙을 고정한다
- 결정: 사용자가 입력한 기준일을 바탕으로 D+N(예: 100일), 월 단위, 연 단위 기념일을 계산한다.
- 결정: 달력에서 기념일은 일반 일정과 구분되는 뱃지/점표시를 사용하고, 제목은 `기념일명 · D+N` 형식으로 노출한다.
- 이유: 하루 단위 사용 맥락에서 "오늘이 무슨 날인지"를 빠르게 인지하게 해야 한다.
- 대안:
  - 수동 입력만 허용: 반복 기념일 누락 가능성이 높다.

## Risks / Trade-offs

- [의미 불명확한 신조어 남용] -> Mitigation: 기본 의미 라벨 병기 및 툴팁 제공
- [화면별 용어 불일치] -> Mitigation: 중앙 용어집 참조 강제 + 리뷰 체크리스트 추가
- [캘린더 제목 길이 초과/가독성 저하] -> Mitigation: 제목 길이 제한 규칙(짧은 라벨 우선) 적용
- [윤년/말일 기준 계산 오차] -> Mitigation: 날짜 계산 규칙(윤년/말일 보정)과 테스트 케이스를 명시
- [유행어 수명 짧음] -> Mitigation: 핵심 기능명은 안정형으로, 이벤트형 문구만 교체 가능하게 분리
- [사용자 취향 편차] -> Mitigation: 향후 기본/햄찌 톤 전환 옵션을 확장 가능 항목으로 남김

## Migration Plan

1. `hamjji-brand-lexicon` 스펙 정의 및 기존 기능 스펙의 문구 변경 포인트를 식별한다.
2. 역할/세션/보상/일정/면접 용어 키 목록을 만든다.
3. UI 문자열 하드코딩 위치를 매핑 테이블 참조 방식으로 교체한다.
4. 캘린더/리포트 템플릿에 키치 용어를 적용하고 길이/가독성 검수 규칙을 반영한다.
5. QA 체크리스트에 금지어/의미 명확성/톤 일관성 항목을 추가해 검증한다.
6. 기념일 계산기 규칙을 유틸로 분리하고 달력 표기 UI와 연결한다.

Rollback 전략:
- 레이블 맵을 기존 기본 용어 세트로 즉시 스위치할 수 있게 유지한다.
- 외부 연동 메시지는 기본 문구 fallback을 항상 유지한다.

## Open Questions

- 역할 용어 최종안: `나햄찌/햄찌메이트` vs 다른 표현 중 어떤 조합이 가장 자연스러운가?
- "데이트데이", "추억카" 외에 고정 채택할 상위 10개 용어를 무엇으로 할 것인가?
- 유행어 반영 주기를 릴리즈 단위로 할지, 콘텐츠 패치 단위로 할지?
- 사용자별 톤 전환 옵션(기본/키치)을 이번 변경에서 명시할지, 후속 변경으로 분리할지?
- 반복 기념일 기본 세트(100일/200일/1주년 등)를 고정 제공할지 사용자 커스텀만 허용할지?
